# ארכיטקטורה אסטרטגית — מנוע זכויות ניידות

## חזון: מפעל מנועי מדיניות ממשלתי

הפרויקט מתוכנן לא רק כמחשבון ניידות, אלא כתשתית לייצור מנועי מדיניות לכל תחום רווחה.

```
מסמכי מקור → חילוץ משתנים → מנוע כללים → מחשבון → ולידציה → ממשל
     ↑                                                              ↓
  תחום חדש ←←←←←←←←← אותו Framework ←←←←←←←←←←←←←←←←←←←←←←←←←←←
```

שיטת עבודה: פקיד מזין ← מנוע ממליץ ← פקיד מחליט ← פידבק ← שיפור

### שלב 2 — ולידציה ובנצ'מרקינג

| שלב | היקף | מטרה |
|-----|------|------|
| 2A | 50-100 מקרים היסטוריים | זיהוי פערים, סתירות, עידון כללים |
| 2B | 500-1000 מקרים היסטוריים | בנצ'מרק מול החלטות פקידים, מדידת דיוק |

מדדים:
- accuracy — אחוז התאמה להחלטות היסטוריות
- false_approvals — אישורים שגויים
- false_denials — דחיות שגויות
- ambiguity_rate — שיעור מקרים שדורשים שיקול דעת
- policy_superiority — מקרים שהמנוע מזהה זכאות שהפקיד פספס

תוצרים: דוחות פערים, מדדי אמון, ציון מוכנות

### שלב 3 — אינטגרציה ארגונית

- פריסה פנימית מאובטחת
- microservice דטרמיניסטי
- אינטגרציה למערכות ליבה
- קליטה אוטומטית של חוזרים חדשים
- אוטומציה מבוקרת (סוג א בלבד)

## מיקום AI

| שלב | תפקיד AI | הגבלה |
|-----|----------|-------|
| נוכחי | אופציונלי — סיוע בקליטה, חילוץ מסמכים | AI לא מחליף לוגיקה דטרמיניסטית |
| עתידי | פרסור מקרים, המלצות, ממשק שיחתי | הליבה תמיד דטרמיניסטית |

כלל ברזל: AI לעולם לא מחליף לוגיקה משפטית דטרמיניסטית.

## ארכיטקטורת מפעל — הרחבה לתחומים נוספים

```
mobility/     ← פעיל
disability/   ← עתידי
long-term-care/ ← עתידי
income-support/ ← עתידי
child-benefits/ ← עתידי
```

כל תחום = אותו framework:
1. מסמכי מקור (PDF)
2. חילוץ משתנים
3. קידוד כללים (RuleLogic.json)
4. עצי החלטה
5. מחשבון + API
6. ולידציה + בנצ'מרק
7. ממשל + פריסה

---

## עקרונות מנחים

1. MVP קודם — ערך תפעולי מיידי
2. אמון לפני אוטומציה — הפקיד מחליט
3. ולידציה לפני אינטגרציה — הוכחת דיוק
4. לוגיקה דטרמיניסטית כליבה — אפס הסתברות
5. AI כהגברה בלבד — לא כמחליף
6. מפעל מדיניות — framework חוזר לכל תחום
7. ממשל תמיד — ביקורת, הסברים, גרסאות
