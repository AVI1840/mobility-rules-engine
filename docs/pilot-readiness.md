# Pilot Readiness — מנוע זכויות ניידות

## סטטוס: POC/MVP — פיילוט ראשוני

---

## מה הכלי עושה

כלי תומך החלטה שמתרגם חוזרים, הסכמים ופסקי דין מתחום הניידות ללוגיקה דטרמיניסטית ומוסברת.

- מקבל נתוני מקרה (אחוז נכות, מצב תפעולי, סוג תביעה)
- מריץ כללים רלוונטיים בלבד לפי סוג התביעה
- מחזיר המלצת החלטה עם הנמקה וציטוט משפטי
- מסווג כל החלטה לפי רמת ודאות (דטרמיניסטי / המלצה / שיקול דעת)
- מסמן מקרים שדורשים שיקול דעת אנושי

## מה הכלי לא עושה

- לא מחליף שיקול דעת מקצועי
- לא מערכת מבצעית — פיילוט בלבד
- לא משתמש בבינה מלאכותית גנרטיבית — לוגיקה דטרמיניסטית בלבד
- לא לומד אוטומטית מהחלטות פקידים
- לא משנה כללים אוטומטית

## הפעלה מקומית

```bash
npm install

# terminal 1 — שרת
cd packages/server && npm run start

# terminal 2 — ממשק
cd packages/client && npm run dev
```

פתח http://localhost:5173

## הרצת בדיקות

```bash
cd packages/server && npx vitest run
```

## בדיקת תרחיש

1. פתח "בדיקת זכאות"
2. הזן: ת.ז. כלשהי, 50% נכות, סמן "שוהה במוסד"
3. לחץ "בדוק זכאות"
4. צפה בתוצאה + הנמקה + ציטוט

## ולידציה מול מקרים היסטוריים

1. פתח "ולידציה"
2. העלה קובץ JSON עם מקרים (ראה sample ב-data/test-cases/)
3. צפה בדוח: דיוק, אישורים שגויים, דחיות שגויות

## מגבלות ידועות

- 14 כללים מקודדים — לא כל הלוגיקה של כל חוזר
- הכללים מבוססים על פרשנות ראשונית — דורשים ולידציה מקצועית
- אין אינטגרציה למערכות ליבה
- אין ניהול מחזור חיי מדיניות
- אין בקרת גישה מבוססת תפקידים
- Audit trail נשמר לקובץ מקומי, לא ל-DB

## מה נדרש לבדיקת היתכנות

1. הצגה למנהלת תחום (15-20 דקות)
2. בדיקת היגיון מקצועי של הכללים
3. מדגם 50-100 מקרים היסטוריים אנונימיים
4. פידבק מפקיד בכיר

## הוספת domain חדש (עתידי)

```
packages/server/src/domains/
  mobility/          ← קיים
  disability/        ← עתידי
  long-term-care/    ← עתידי
```

כל domain חדש דורש:
1. קבצי כללים (RuleLogic.json) תחת `domains/{name}/rules/`
2. קטלוג משתנים תחת `domains/{name}/variables/`
3. module.ts שטוען את הכללים
4. רישום ב-server.ts: `engine.loadDomainModule(newModule)`
